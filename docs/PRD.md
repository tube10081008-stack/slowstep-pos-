# 📋 PRD — 슬로우스텝 멤버십 POS

## 1. 배경 & 문제

슬로우스텝은 결제사 **payhere** 의 프로그램으로 결제·멤버십을 운영해 왔다.
직접 모은 멤버십 고객이 약 **220명**에 달하지만, 이들에게 마케팅(쿠폰·재방문 유도·세그먼트 발송)을 하려면
결제사의 **고객관리 프리미엄 기능을 유료 구매**해야 한다.

> **핵심 통증:** 내가 모은 고객 데이터인데도, 결제사 정책·과금에 종속되어 자유롭게 활용할 수 없다.

## 2. 목표 (Goals)

1. **고객 데이터 소유권 회복** — 회원·거래·적립 데이터를 우리 DB에 보관, 결제사 동의 없이 마케팅 활용.
2. **결제 종속 탈피** — Toss페이먼츠 + 단말기로 자체 결제 처리.
3. **재방문 유도** — 게이미피케이션(스탬프·등급·미션)으로 LTV 상승.
4. **현장 운영성** — 직원이 스마트폰 한 대로 결제·적립·조회를 끝내는 모바일 POS.

### Non-Goals (이번 범위 밖)

- 실제 카드 단말 하드웨어 드라이버 (P1)
- 회계/세무 정산 자동화 (P3)
- 멀티 매장/프랜차이즈 (P3)

## 3. 사용자 & 시나리오

| 역할 | 니즈 | 핵심 화면 |
| --- | --- | --- |
| **직원(바리스타)** | 빠른 결제·적립, 회원 식별 | 모바일 POS |
| **고객(멤버)** | 내 포인트/등급/미션 확인, 결제 적립 | QR 멤버십 페이지 |
| **점주** | 고객 현황·마케팅 대상 파악 | 관리자(Django admin) → P2 대시보드 |

### 대표 플로우

**A. 매장 결제 + 적립**
1. 고객이 자신의 QR(회원번호=연락처) 제시 → 직원 POS가 스캔/입력으로 회원 식별
2. 직원이 주문 금액 입력 → (선택) 보유 포인트 사용
3. Toss로 결제 승인 → 거래 확정
4. 결제 금액 기준 **포인트 자동 적립**, 방문 스탬프 +1, 등급/미션 갱신

**B. 고객 셀프 조회**
1. 고객이 매장 QR 또는 본인 QR을 스캔 → 멤버십 페이지 진입
2. 회원번호(연락처)로 본인 식별 → 포인트·등급·스탬프·진행 중 미션 확인

## 4. 기능 요구사항

### 4.1 회원 (Member)
- 회원번호 = **연락처(휴대폰)**. 가입 시 이름·연락처만으로 즉시 생성.
- 보유 포인트, 누적 결제액, 방문 횟수, 등급, 가입일 보관.
- QR 페이로드 = 회원번호(연락처). 조회/적립의 키.

### 4.2 거래 (Transaction)
- 결제 1건 = 거래 1건. 총액·사용 포인트·실결제액·적립 포인트·결제수단·Toss 키 보관.
- 상태: `pending`(승인 전) → `paid`(승인) → `canceled`(취소/환불).

### 4.3 포인트 (Point)
- 기본 적립률 **5%** (실결제액 기준, 설정값). 결제 확정 시 적립.
- 사용은 1P=1원, 결제 시 차감. 모든 증감은 **원장(PointEntry)** 에 기록 → 잔액 추적·감사.

### 4.4 게이미피케이션
- **스탬프:** 방문(결제)마다 +1, N개 모이면 리워드(설정).
- **등급:** 누적 결제액 기준 `BRONZE → SILVER → GOLD`. 등급별 적립 보너스(향후).
- **미션:** "이번 달 3회 방문", "누적 5만원" 등 조건 달성 시 보너스 포인트. `Mission`/`MemberMission`로 진행률 추적.

### 4.5 결제 (Toss)
- 단말/온라인 결제 승인을 Toss페이먼츠로 처리. 상세는 [`TOSS-INTEGRATION.md`](./TOSS-INTEGRATION.md).
- 이번 스캐폴드: 승인 흐름과 데이터 모델·서비스 인터페이스까지. 실 키 연동은 환경변수 주입 시 동작.

## 5. 성공 지표 (KPI)

- payhere 프리미엄 미구매로 **마케팅 가능 회원 100%** 확보(소유권).
- 재방문율 / 회원당 월 방문수 상승 (게이미피케이션 효과).
- 적립·사용 정상 처리율, 결제 승인 성공률.

## 6. 로드맵

| 단계 | 내용 |
| --- | --- |
| **P0 (현재)** | 데이터 모델 · 회원/적립/거래 API · Toss 연동 설계 · 동작하는 웹 POS·QR 골격 · 데모 시드 |
| **P1** | Toss 실결제 승인·웹훅, 단말 페어링, 거래 취소/환불, 정산 리포트 |
| **P2** | 미션 엔진·푸시 알림, 마케팅 세그먼트·발송(문자/알림톡), 점주 대시보드 |
| **P3** | 멀티 매장, 회계/정산 연동, 분석 대시보드 |

## 7. 리스크 & 고려사항

- **개인정보:** 연락처가 회원 식별자 → 수집·이용 동의, 암호화/접근통제 필요(P1 강화).
- **PG 심사:** Toss 가맹 심사·정산주기. 본인인증/현금영수증 정책 확인.
- **QR 보안:** 회원번호 노출 최소화 — 단순 연락처 대신 토큰형 QR 검토(P1).
- **오프라인:** 네트워크 단절 시 결제/적립 큐잉 전략(P2).
